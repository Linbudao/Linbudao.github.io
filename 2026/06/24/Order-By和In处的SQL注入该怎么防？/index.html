Order By和In处的SQL注入该怎么防？

标题的问题来自学长总结的一道高频面试问题

当时看到的时候，觉得还是挺简单的，因为Order by 和In后面跟的东西本质就不一样

但是为了做到尽量满分不漏回答，在读了一些文章配合ai深入学习之后记录一下

依旧要想知道怎么防御就要知道其漏洞原理

In处

这里我想先说说In，因为in和where id=是有异曲同工之妙的，本质都是“值位”的参数绑定场景

where id=是经典的sql注入点

预编译正是为了防御此类sql注入

这个机制放到 IN 上也是一样的，只是 IN 是可变长度的参数列表

很显然，通过预编译，语法结构被锁死，in后的三个?里面待传的也只是数据罢了

这就是预编译的特点：结构先定型，数据后填充，数据无法反向污染结构

所以In处的防御核心其实也是实现参数化查询/预编译

Order by处

本文的饺子醋就是order by

预编译中，值value是可以用占位符去替代的，但是标识符不能用占位符代替

比如字符串admin，数字123这些是数据本身，而标识符是数据库对象的名字，比如表名users列名id等

这些位置是SQL查询结构的一部分，若将这些位置交给用户控制输入则控制的是查询语义而不再是数据值

order by的查询后面跟的只能是以下几种

  • 标识符
  • 位置序号
  • 函数调用
  • CASE表达式

第三四点合为任意可排序表达式

Order by注入

确认存在&怎么确定是order by

有些时候黑盒光看参数特征无法知道参数所在的具体sql语法结构，并且大多数时候报错也不会回显

ASC/DESC测试

这是order by的语法特性，只在order by的语境下有意义，可以用这个作探测

而在where的语法结构中，DESC没有意义也不影响结果

Order by独特的数字语义

order by后面直接跟数字，其结果集的排序随之改变，其数据量不变

比如sort=1按id排列，sort=2按Name排列，sort=3按Price排列

并且在数值超出列的最大值时有可能会报错

而在where id=的结构中，id后面的数字影响的是数据的完全变化

盲注里面的体现

基础结构

很好读懂，条件为真就按列A排序，条件为假就按列B排列

正是因为这一点，开发者不得不拼接这些部分

在代码中的体现

而上面我们知道，order by后面也是可以跟数字的，在跟数字的时候为什么不能用预编译防不了呢？

原因是order by和where id=后面跟的数字身份不一样

在where结构中，id后面的数字是一个值节点，一个用来比较的数据

而在order by中，后面的数字是一个排序项序号引用，本质还是去找数值对应的列

这样又回到了最开始的问题，预编译的只能是数据而不能是数据库本身的对象

其实允许用户输入直接作为列名和表名本身就是开发设计上的缺陷，但有很多不得不保留而不考虑完全重写的情况

最好的方案就是进行白名单映射

白名单映射

旨在永远不让用户输入的直接进入sql

白名单映射，用户输入的必须在映射表里面，如果不在就会返回默认值

这个挺好理解的，其实和白名单很像，只是前端负责了这个过程

最小权限/权限分级原则

在数据库层面划分权限，普通业务账号设置只读或者限制功能，这样顶多是数据泄露而不是删库（drop权限）等改的操作

总结

参数化查询主要保护的是数据值上下文。像列名、表名、排序方向、表达式片段、SQL 关键字等结构性内容，通常不能直接靠占位符安全替代。一旦让用户输入直接决定这些结构元素，就需要白名单映射或重构查询逻辑
